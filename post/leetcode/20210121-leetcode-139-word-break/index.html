문제

Given a non-empty string s and a dictionary wordDict containing a list of non-empty words, determine if s can be segmented into a space-separated sequence of one or more dictionary words.

Note:

  • The same word in the dictionary may be reused multiple times in the segmentation.
  • You may assume the dictionary does not contain duplicate words.

기본 테스트 케이스

Example 1:

1Input: s = "leetcode", wordDict = ["leet", "code"]
2Output: true
3Explanation: 
4  - Return true because "leetcode" can be segmented as "leet code".

Exapmle 2:

1Input: s = "applepenapple", wordDict = ["apple", "pen"]
2Output: true
3Explanation: 
4  - Return true because "applepenapple" can be segmented as "apple pen apple".
5  - Note that you are allowed to reuse a dictionary word.

Exapmle 3:

1Input: s = "catsandog", wordDict = ["cats", "dog", "sand", "and", "cat"]
2Output: false

핵심 아이디어 (백트래킹)

  • 문제 살펴보기

    • 비지 않은 String s와 비지 않은 단어들을 담긴 List들을 가진 wordDict가 메서드의 인자로 주어진다.
    • 문자열 s가 wordDict 내의 단어들로 깔끔하게 다 쪼개지는지 판단하는 문제이다.
    • 문자열 s를 쪼개기 위해 wordDict 내의 단어들을 얼마든지 재사용할 수 있다.
    • wordDict 내에는 중복되는 단어가 없다.

  • 아이디어

    • 문자열 s의 0번째 인덱스부터 모든 단어(wordDict 내의 단어들)를 비교해가며 DFS 방식으로 탐색한다.
    • 만약 탐색 불가능한 경로의 경우 HashMap에 등록한다.
    • 특정 경로로 진입했을 때 이전에 방문했던 경로라면 더 이상 탐색하는 것이 의미 없으므로 해당 경로의 탐색을 중단한다. (가지치기)
    • 자세한 설명은 생략한다. 아래 두 예시를 확인하자.

두 번째 예시 (답을 찾지 못할 때)

  • 실수
    • 처음에 백트래킹 방식이 아닌 단순 DFS로 완전 탐색 했다.
    • 이 경우 아래와 같은 테스트 케이스에서 TLE가 발생한다.
    • 이를 해결하기 위해 이미 한 번 방문했던 경로를 기록해뒀다가 매번 특정 경로에 진입할 때 마다 방문 이력이 있는 경로인지 확인함으로써 최적화 해야 할 필요성을 느껴 backtracking 기법을 적용했다. 
      1Last executed input:
2"aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaab"
3["a","aa","aaa","aaaa","aaaaa","aaaaaa","aaaaaaa","aaaaaaaa","aaaaaaaaa","aaaaaaaaaa"]
    • 그래서 HashMap을 활용해 메모이제이션 역할을 수행하도록 하였다.
    • 다음부터는 DFS 방식의 완전 탐색 시에 항상 최적화(backtracking 적용)를 염두해 두어야 겠다.

구현 코드 (백트래킹)

 1class Solution {
 2    
 3    public boolean wordBreak(String s, List<String> wordDict) {
 4        return backtrack(0, s, s.length(), wordDict, new HashMap<String, Boolean>());
 5    }
 6    
 7    private static boolean backtrack(int idx, String s, int len, List<String> wordDict, HashMap<String, Boolean> failResStrMap) {
 8        // 이미 한 번 거쳤던 경로 라면? return false
 9        if (failResStrMap.containsKey(s.substring(idx))) {
10            return false;
11        }
12        
13        boolean ret = false;
14        if (idx == len) { // 문자열이 완벽히 다 쪼개졌다면? return true (정답)
15            return true;
16        } else {
17            // 완탐 (dfs)
18            for (String word : wordDict) {
19                if (idx+word.length() <= len && s.substring(idx, idx+word.length()).equals(word)) { // 다음에 탐색할 문자열 범위가 유효하다면? 그리고 다음에 탐색할 문자열이 리스트 내에 존재한다면? 탐색 시작
20                    if (backtrack(idx+word.length(), s, len, wordDict, failResStrMap)) { // backtrack() 함수 호출 결과가 true라면 정답을 찾은 것이므로 탐색 종료
21                        ret = true;
22                        break;
23                    }
24                }
25            }
26        }
27
28        // 해당 경로로 탐색 했으나 정답을 찾지 못했다면?(ret == false인 경우)
29        // 가지치기를 위해서 해당 경로를 fail 경로(재탐색하면 안되는 경로)로 등록
30        if (!ret) {
31            failResStrMap.put(s.substring(idx), false);
32        }
33
34        return ret;
35    }
36}

아이디어 (DP) (feat. 틀림, 자세한 건 아래에서 설명)

  • dp
    • dp[i] == true 의미
      • 문자열 s의 substring(0~i)을 완벽히 다 쪼갤 수 있음을 의미
    • 점화식
      • dp[i] = dp[t] && wordSet.contains(s.substring(t, i))
      • sub problem: dp[t]
      • super problem: dp[i]
    • 시간 복잡도 O(n+k)로 풀 수 있다
      • n: 문자열의 길이
      • k: 단어의 개수


구현 코드 (DP)

  • 다른 사람의 코드(O(n^2) 시간 복잡도)를 O(n+k)로 최적화 시켜 보았다.
 1class Solution {
 2    public boolean wordBreak(String s, List<String> wordDict) {
 3        
 4        HashSet<String> set = new HashSet<>();
 5        for (String word : wordDict) {
 6            set.add(word);
 7        }
 8        
 9        boolean[] dp = new boolean[s.length()+1];
10        int t = 0;
11        dp[t] = true;
12        for (int i = 1; i < s.length()+1; i++) {
13            if (dp[t] && set.contains(s.substring(t, i))) {
14                dp[i] = true;
15                t = i;
16            } else {
17                dp[i] = false;
18            }
19        }
20            
21        return dp[s.length()];
22    }
23}
  • 근데 아래의 테스트 케이스에서 답을 도출하는데 실패했다.
1"aaaaaaa"
2["aaaa","aaa"]
  • 이유는 앞에서 부터 길이가 짧은 substring부터 set에 존재하는지 확인하기 때문에 이 경우는 "aaa_aaa_a"로 분할된다.
  • "링크에서 제시된 코드를 최적화 할 수 있다"라고 생각했다.
  • 생각한건 문제가 되지 않는다.
  • 문제는 그 이후의 나의 행동이다.
  • 코드를 최적화 시킬 수 있다고 확신하고 블로그에 업로드 할 ppt를 먼저 만드는게 아니라 아이디어가 유효한지 검증하는 과정이 우선됐어야 했다.
  • 이건 굉장히 안좋은 습관이다. 내가 가진 생각, 아이디어를 나 스스로 의심하지 않고 있다는 방증이다.
  • 이러한 습관은 단순한 시간 낭비 뿐 만 아니라 어떤 경우는 나 혹은 누군가에게 굉장히 크리티컬한 영향을 미칠 수 있다는 사실을 명심하자.
  • 다시 본론으로 돌아가자면...
  • 결국은 가장 긴 substring 부터 set에 존재하는지 확인하기 위해 O(n^2)으로 구현해야 한다.
  • 그게 아니라면 문자열 맨 앞에서부터 매번 가장 길이가 긴 문자열 부터 차례로 비교해주는 방식으로 구현할 수도 있다. 이때 시간 복잡도는 O(n*k)가 되며 k가 n보다 클 경우 O(n^2) 보다 느리다.
  • 다음부터는 ppt 먼저 만들지 말고 내가 생각한 아이디어가 유효한지 코드 부터 짜서 돌려보고 정답을 도출할 수 있는 아이디어라면 그때 ppt로 만들어야 겠다. ㅠㅠㅠ
  • 그래도 좋은 시도였다. 막연히 "O(n) 솔루션으로 풀 수 있는데..." 라고 생각만 했다면 이런 경우의 테스트 케이스를 통과하지 못한다는 사실을 인지하지 못했을 테니까 말이다.

후기

  • 사실 이 문제를 푼 이유는 DP 유형을 연습하기 위해서였다.
  • 그러나 DP로는 해결 방법이 보이지 았았다.
  • 근데 DFS 방식으로 구현하면 답을 도출할 수 있을 것 같아 DFS로 구현했고 TLE가 발생했다.
  • 원인은 이미 한 번 방문했던 경로도 계속해서 반복 탐색하기 때문에 비효율적이였다.
  • 그래서 이미 한 번 탐색을 진행했으나 단어를 완벽히 쪼갤 수 없었던 경로를 HashMap에 저장했다. (지금 생각해보니 특정 경로로 진입했을 때의 문자열만 저장하면 되니 HashMap이 아닌 HashSet이 적합할 듯 하다.)
  • 그리고 매번 특정 경로에 진입할 때 마다 이미 방문 이력이 있던 경로인지 확인하고 만약 이미 방문했다면 더 이상 해동 경로를 탐색하지 않도록 backtracking 기법을 적용함으로써 TLE를 해결했다.
  • 문제를 해결한 후 다른 사람들의 DP 솔루션을 보니 사고 방식이 신기했다.
  • 나는 문자열 s를 단어 단위, 즉 sub word 단위로 쪼갤려고 생각했는데 다른 사람들은 하나의 문자 단위로 쪼개서 super problem과 sub problem으로 나누고 점화식을 도출한 것이다.
  • 이는 내가 생각할 수 있는 기존의 사고 방식과는 거리가 멀어서 신선한 충격을 받았다.
